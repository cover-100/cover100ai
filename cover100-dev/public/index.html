The report never lies to you

Coverage tools love a green number. cover100’s contract is stricter: every package ends a run in exactly one of three states — and each state carries its receipts.

100% verified

Fully covered, with tests an independent verifier accepted: they build, they pass, they assert real behavior — coverage-padding tests (_ = result, calls without checks) are rejected, not counted.

Documented gap

A branch that genuinely can’t be covered without refactoring gets an entry in the package’s TEST-COVERAGE.md: why it’s un-coverable, and the exact refactor that would unlock it. Settled gaps are never silently re-fought on the next run — and never silently dropped.

Quarantined

Packages that don’t build, generated code, vendored trees — listed with reasons, excluded from the claim. The tool degrades by telling you, never by pretending.

How it works

A deterministic pipeline of specialized agents — not one heroic prompt.

  1. Collect

    A deterministic script measures the statement-coverage baseline and triages the work: already-covered, settled-gap and broken packages are skipped with zero spend.

  2. Research

    Researchers classify why each region is uncovered — error path, time/random, external I/O, defensive code — and prescribe the concrete testing approach before any test is written.

  3. Engineer

    Parallel engineers work in isolated git worktrees, one unit each, committing every package the moment it’s green. Shared fakes live in a keyed knowledge base so the same mock is never invented twice.

  4. Verify

    An independent safety verifier gates every unit: build green, assertions meaningful, production changes limited to declared seams, no new lint findings. Unverified work doesn’t merge.

  5. Integrate & report

    Serialized merges, a whole-module finalize pass, and a portable REPORT.md: per-package table, gap register, dependency list, and instructions the next run consumes automatically.

No work lost. Ever.

Abort a run mid-flight, lose your connection, kill an agent — finished packages are already committed, interrupted units are parked, and the next run salvages everything that builds and passes. Resuming is the default motion, not a recovery procedure.

“Verified” is doing real work in that sentence

Generation is cheap; the harness is the product. Things the safety verifier rejected in real runs:

Coverage padding

_ = w.Code after an elaborate handler setup — ten tests in one package executed code without asserting anything. Dropped from the merge, named in the report, line by line.

Production “fixes”

The only production change a test run may make is an explicit, reviewable seam (var now = time.Now). A helper file smuggled outside the assigned package? Rejected at 98.8% coverage.

Give-up documentation

A “can’t cover this” entry that a known pattern could cover gets flagged in the report and re-targeted — documented gaps are challenged, not taken on faith.

Tests are executable specs — writing them finds bugs

Driving code to 100% means exercising the branches nobody ever ran. Real finds from early runs: an inverted boolean method, a goroutine silently discarding errors by checking the wrong variable, a field appended from its sibling field, data races in init paths. All captured as explicitly-unverified leads in CODE-REVIEW-NOTES.md — never “fixed” behind your back.

Two modes

drive-to-100

Take a subtree or a whole repository to the claim above. Start small — /cover100 ./pkg/auth/... finishes in one sitting with a cost projection up front; whole-repo runs scale the same harness with budget controls, model escalation for stubborn packages, and cross-run memory.

rescan

Your AI wrote slop tests? rescan audits existing suites: hand-rolled fakes that duplicate the standard library, copy-paste test functions that should be one table test, dead helpers. Then refactors them — coverage may not drop, assertions may not weaken, test code should shrink.
